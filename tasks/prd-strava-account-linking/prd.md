# Product Requirements Document (PRD): Strava Account Linking

## Overview

Sudolife users need a secure way to connect and disconnect their Strava account so Sudolife can identify the athlete associated with the current user and prepare for future Strava-based features. This first version only covers account linking, account ownership validation, token lifecycle readiness, and unlinking. It does not import activities, subscribe to webhooks, or expose Strava activity data.

The feature is for authenticated Sudolife users who already have a Strava account and want to authorize Sudolife to recognize that account. The value is establishing a trusted one-to-one relationship between a Sudolife user and a Strava athlete before any future activity synchronization is added.

## Goals

- Allow an authenticated Sudolife user to start Strava account linking from Sudolife.
- Complete linking only after Strava authorization succeeds and the returned athlete can be uniquely associated with the current Sudolife user.
- Prevent a single Strava athlete from being linked to more than one Sudolife user.
- Allow users to unlink their Strava account from Sudolife.
- Persist enough authorization data to support future token refresh without asking the user to reconnect unnecessarily.
- Track linking success rate, linking failure reasons, unlink events, and duplicate-athlete rejection events.

## User Stories

- As an authenticated Sudolife user, I want to request a Strava authorization link so that I can connect my Strava account from Sudolife.
- As an authenticated Sudolife user, I want Sudolife to complete the connection after I approve access in Strava so that my Sudolife account is linked to my Strava athlete profile.
- As an authenticated Sudolife user, I want to see whether my Strava account is linked so that I understand my current connection state.
- As an authenticated Sudolife user, I want to disconnect Strava from Sudolife so that Sudolife no longer has access to my Strava authorization.
- As a Sudolife user, I want Sudolife to reject linking if my Strava athlete is already connected to another Sudolife user so that my Strava identity cannot be claimed by multiple accounts.
- As a Sudolife operator, I want duplicate-link and authorization failures to be observable so that support and debugging are possible without exposing sensitive tokens.

## Key Features

1. Backend-generated Strava authorization URL

Sudolife must provide an authenticated action that returns a Strava authorization URL for the current user. The URL must direct the user to Strava's OAuth authorization flow and include the required values for client identification, redirect destination, authorization response type, requested scope, and state protection.

Functional requirements:

1. The system must return a Strava authorization URL only for authenticated Sudolife users.
2. The authorization URL must be generated by the backend.
3. The authorization request must support the minimum Strava permissions required to link and unlink the account.
4. The authorization request must include a state value that allows Sudolife to validate the callback belongs to the initiating user.
5. The system must not mark the account as linked when only the authorization URL is generated.

2. Strava authorization callback

Sudolife must handle the user returning from Strava after authorization. The callback must validate the request, exchange the authorization code with Strava, inspect the granted authorization data, and associate the returned Strava athlete with the current Sudolife user.

Functional requirements:

1. The system must reject callbacks with missing, invalid, expired, or mismatched state.
2. The system must exchange the Strava authorization code for authorization data using Strava's OAuth token endpoint.
3. The system must persist the Strava athlete identifier linked to the current Sudolife user.
4. The system must persist token expiration data and refresh authorization data needed for future refresh.
5. The system must validate that the granted Strava scope is sufficient for account linking before completing the link.
6. The system must not expose Strava access tokens or refresh tokens in API responses.
7. The system must return a clear success response when linking is completed.
8. The system must return a clear failure response when Strava authorization is denied, invalid, expired, or insufficient.

3. One-to-one Strava athlete ownership

Each Strava athlete may be linked to at most one Sudolife user. This prevents account confusion, duplicate ownership, and unauthorized reuse of the same Strava identity across multiple Sudolife accounts.

Functional requirements:

1. The system must reject linking when the Strava athlete is already linked to a different Sudolife user.
2. The system must allow an already-linked user to reconnect the same Strava athlete to refresh stored authorization data.
3. The system must preserve the one-to-one relationship under concurrent linking attempts.
4. The system must provide a user-safe error when a Strava athlete is already linked elsewhere.

4. Linked account status

Sudolife must expose the current user's Strava link status so the application can display whether the account is connected.

Functional requirements:

1. The system must identify whether the current Sudolife user has a linked Strava account.
2. The status must include non-sensitive Strava account information needed for display, such as the Strava athlete identifier.
3. The status must not include Strava access tokens, refresh tokens, client secrets, or other sensitive authorization values.

5. Unlink Strava account

Sudolife must allow an authenticated user to disconnect their linked Strava account. After unlinking, Sudolife must no longer treat the user as connected to Strava.

Functional requirements:

1. The system must allow only the authenticated owner to unlink their Strava account.
2. The system must revoke or invalidate Sudolife's Strava authorization when possible.
3. The system must remove the active association between the Sudolife user and the Strava athlete.
4. The system must delete or render unusable stored Strava authorization data after unlinking.
5. The system must treat repeated unlink requests for an already-unlinked account as a safe, user-understandable outcome.

6. Token refresh readiness

Although this version will not import Strava data, Sudolife must store and manage authorization metadata in a way that supports future token refresh.

Functional requirements:

1. The system must store token expiration data returned by Strava.
2. The system must store the latest refresh authorization data returned by Strava.
3. The system must account for Strava refresh token rotation by replacing stored refresh authorization data when Strava returns a new value.
4. The system must not attempt activity synchronization as part of this feature.

## User Experience

The primary user is an authenticated Sudolife user managing external account connections. The expected flow starts inside Sudolife, where the user chooses to connect Strava. Sudolife requests a backend-generated authorization URL and sends the user to Strava. After the user approves access, Strava redirects back to Sudolife and the user receives a clear connected or failed state.

If linking fails because the Strava account is already linked to another Sudolife user, the user must receive a clear message that the Strava account cannot be connected to this Sudolife account. The message must not reveal personal details about the other Sudolife user.

If the user disconnects Strava, Sudolife must show the account as unlinked after the operation completes. Unlinking should be accessible from the same account connection area where the linked status is displayed.

The experience must be accessible through keyboard navigation and screen readers. Status messages for successful linking, failed linking, and unlinking must be perceivable without relying only on color.

## High-Level Technical Constraints

- The integration must use Strava OAuth authorization code flow.
- The backend must generate the Strava authorization URL.
- The callback must exchange the authorization code through Strava's OAuth token endpoint.
- Strava authorization responses include short-lived access tokens, refresh authorization data, expiration metadata, granted scopes, and athlete information.
- Strava refresh authorization data may rotate, so Sudolife must be able to replace stored refresh data when a new value is returned.
- Strava deauthorization must be used during unlinking when a valid authorization is available.
- Strava credentials, access tokens, and refresh tokens are sensitive and must never be exposed to clients or logs.
- The feature must preserve the existing Sudolife authentication model and require an authenticated Sudolife user for all account-management operations.
- A Strava athlete must have a unique active association with only one Sudolife user.
- The feature must comply with the project's hexagonal architecture, boundary DTO, persistence, migration, and testing rules.

## Out of Scope

- Importing Strava activities.
- Synchronizing historical activities.
- Subscribing to or handling Strava webhooks.
- Reading private activity data.
- Displaying Strava activities, segments, routes, clubs, or performance metrics.
- Admin tooling for manually linking or unlinking users.
- Mobile deep-link authorization flows.
- Background jobs for activity sync.
- User-facing analytics based on Strava data.
